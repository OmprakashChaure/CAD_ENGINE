## What This System Does

CAD_ENGINE reads DXF engineering drawings, extracts geometric and semantic structure through a
seven-phase deterministic pipeline, and emits fine-tuning tasks in instruction-tuning JSONL
format. Each task presents a language model with visible engineering context — a feature type,
neighbor geometry dimensions, symmetry relationships, concentric bore structure — and asks it
to infer one missing engineering property: a bore diameter, wall thickness, thread size, or
feature spacing.

The pipeline is fully deterministic. No neural networks, no probabilistic inference, no LLM
calls occur during processing. Given the same DXF file and the same code, every output file is
identical on every run.

The generated dataset is intended to fine-tune Qwen2.5-Coder-7B-Instruct using LoRA on a
Colab T4 GPU. That fine-tuning step has not been implemented yet.

## Seven-Phase Pipeline

### Architecture Overview

```
DXF file  (data/raw_dxf/*.dxf)
    |
    v  Phase 1   ExtractionPipeline        pipeline/extraction_pipeline.py
    |             Load -> iterate -> normalize -> filter (5 filters)
    |             Output: entities[], quarantined_entities[], filter_reports[]
    |
    v  Phase 2   TopologyPipeline          pipeline/topology_pipeline.py
    |             Vertex indexing -> adjacency -> orphan detection
    |             Output: shared_vertices, adjacency_list, orphan_entities
    |
    v  Phase 3   StructuralPipeline        pipeline/structural_pipeline.py
    |             Contours -> loops -> concentric groups -> regions -> contour hierarchy
    |             Output: loops, concentric_groups, regions, contour_hierarchy
    |
    v  Phase 4   FeaturePipeline           pipeline/feature_pipeline.py
    |             Hole candidates -> slot candidates -> radial patterns
    |             -> symmetry -> feature-region grouping
    |             Output: hole_candidates, slot_candidates, radial_patterns, symmetries
    |
    v  Phase 5   RefinementPipeline        pipeline/refinement_pipeline.py
    |             Confidence scoring -> hierarchy -> conflict detection
    |             -> repetition consolidation -> ambiguity tracking
    |             Output: refined candidates, repetition_groups, ambiguity flags
    |
    v  Phase 6   ContextPipeline           pipeline/context_pipeline.py
    |             Relationship building -> cluster analysis -> confidence scoring
    |             -> structural dependency mapping -> ambiguity propagation
    |             Output: relationships, clusters
    |
    v  Phase 7a  DatasetPipeline           pipeline/dataset_pipeline.py  (DatasetPipeline class)
    |             Per-drawing: supervision mapping, context packaging,
    |             target construction, inference conditioning, sample assembly
    |             Output: data/intermediate/{run_id}/phase7_dataset/{id}_dataset.json
    |
    v  Phase 7b  DatasetExporter           pipeline/dataset_pipeline.py  (DatasetExporter class)
                  All drawings: SemanticPipeline -> semantic records
                  -> task generation -> dedup -> quality filter
                  -> balance -> deterministic split -> JSONL write
                  Output: data/intermediate/{run_id}/phase7_export/
                          train.jsonl, validation.jsonl, test.jsonl
                          metadata.json, semantic_records.json
                          semantic_metadata.json, semantic_coverage_audit.json
```

The orchestrator is `main.py`. It loops over every `*.dxf` in `data/raw_dxf/`, runs phases
1-7a per file, accumulates results in memory, then calls `DatasetExporter.export()` once on
the entire batch.

### Phase 1: Extraction and Filtering

**Module:** `pipeline/extraction_pipeline.py`

`ExtractionPipeline` loads a DXF file via `DXFLoader`, iterates the modelspace via
`EntityIterator`, normalizes each entity through `GeometryNormalizer`, then passes the entity
list through five sequential filters.

[source: `pipeline/extraction_pipeline.py`, full file]

#### DXF Loading

`core/reader/dxf_loader.py`, `DXFLoader.load()`:

1. Validates that the file exists and has a `.dxf` extension.
2. Calls `ezdxf.readfile()`.
3. On any exception, falls back to `ezdxf.recover.readfile()`.
4. If recovery also fails, raises `RuntimeError`.

[source: `core/reader/dxf_loader.py`, full file]

**Note:** `core/reader/dxf_reader.py` does not exist. The prior documentation listed it, but
it was never created. The reader is `dxf_loader.py` + `entity_iterator.py` only.

#### Entity Types

Recognized entity types (constant `SUPPORTED_TYPES` in `entity_iterator.py`):

[source: `core/reader/entity_iterator.py`, L15-27]

```
LINE, CIRCLE, ARC, LWPOLYLINE, POLYLINE, SPLINE, TEXT, MTEXT, INSERT, DIMENSION, HATCH
```

Any DXF entity type not in this set is silently skipped.

Entities that ARE in this set but lack supported geometry — SPLINE, HATCH, INSERT, and any
TEXT/MTEXT/DIMENSION that `GeometryNormalizer` cannot parse — receive `supported: False` in
their normalized dict. They are quarantined by `DegenerateFilter` rather than deleted, and
remain accessible in the pipeline output for potential future annotation extraction.

**Entity identity** is assigned as `ent_{counter:05d}` where `counter` is a sequential integer
per DXF file starting at 1.

[source: `core/reader/entity_iterator.py`, L86]

#### Geometry Normalization

`core/classifiers/geometry_normalizer.py`, `GeometryNormalizer.normalize()`:

[source: `core/classifiers/geometry_normalizer.py`, full file]

| Entity type | Canonical fields produced |
|-------------|---------------------------|
| LINE | `start [x,y]`, `end [x,y]`, `length` |
| CIRCLE | `center [x,y]`, `radius`, `diameter`, `area` |
| ARC | `center [x,y]`, `radius`, `start_angle`, `end_angle` |
| LWPOLYLINE | `points [[x,y]...]`, `closed`, `has_arcs`, `arc_segments` (per-vertex bulge data) |
| POLYLINE | `points [[x,y]...]`, `closed`, `has_arcs`, `arc_segments` |
| DIMENSION | `dimension_type`, `value`, `text`, `position`, `target_points` |
| TEXT / MTEXT | `text`, `position`, `height`, `text_role`, `numeric_value` |

**DIMENSION value extraction** (in priority order):
1. `parse_fractional_value()` on raw text — handles `1/2`, `3/4`, `1 1/4`, etc.
2. `entity.dxf.actual_measurement` attribute if positive.
3. Regex extraction from stripped text: removes leading `NxN` count notation, extracts first
   numeric token.

**TEXT `text_role` classification:**
`thread_callout` | `tolerance` | `radius_value` | `diameter_value` | `angle_value` |
`dimension_value` | `annotation`

#### Filter Chain

Applied in order; each filter receives the kept entities from the prior filter:

1. **`TextFilter`** — quarantines TEXT, MTEXT, DIMENSION entities (annotation path).
2. **`DegenerateFilter`** — removes zero-length lines, zero-radius circles, and entities with
   `supported: False` (these go to `quarantined_entities`, not `removed_entities`).
3. **`DuplicateFilter`** — removes geometrically identical entities by exact coordinate match.
4. **`LayerFilter`** — reads `configs/layer_rules.yaml`; removes entities on layers in
   `ignore_layers` list (`DEFPOINTS`, `VIEWPORT`, `VIEW`, `TEXT`, `ANNOTATION`, `TITLE`,
   `BORDER`). **This is the only place any YAML config is read at runtime.**
5. **`BorderFilter`** — detects and removes the drawing border/title block rectangle.

 `core/filters/layer_filter.py`, L22-28;
 `configs/layer_rules.yaml`, full file]

### Phase 2: Topology Graph Construction

**Module:** `pipeline/topology_pipeline.py`

[source: `pipeline/topology_pipeline.py`, full file]

Three stages, always in sequence:

**Stage 1 — VertexIndexer** (`core/grouping/vertex_indexer.py`):

Extracts endpoints from LINE, ARC, POLYLINE, LWPOLYLINE. Circles do not contribute endpoints —
their centers are not vertices in the topology graph. Rounds coordinates to 4 decimal places
(`VERTEX_PRECISION = 4`) for deterministic snapping. Produces `vertex_map`:
`(rounded_x, rounded_y)` -> list of entity_ids sharing that vertex.

[source: `core/grouping/vertex_indexer.py`, L25, L36]

**Stage 2 — AdjacencyBuilder** (`core/grouping/adjacency_builder.py`):

From `shared_vertices`, builds a deduplicated, direction-independent `adjacency_list`. Vertices
with more than `MAX_HUB_SIZE = 8` connected entities are treated as noise hubs (typical at
origin points or heavily shared corners) and skipped — their connections are not added to the
adjacency list.

[source: `core/grouping/adjacency_builder.py`, L23]

**Stage 3 — Orphan detection** (`topology_pipeline.py`, `_detect_orphans()`):

Entities with no entry in `adjacency_list` are flagged as `orphan_entities`. Circles are
always orphans because their center is not a topology vertex.

Outputs: `shared_vertices`, `edges`, `adjacency_list`, `orphan_entities`, `statistics`.

### Phase 3: Structural Recognition

**Module:** `pipeline/structural_pipeline.py`

Five sub-stages:

1. **Contour extraction** (`core/grouping/contour_extractor.py`): traverses the adjacency graph
   to find maximal chains of connected entities. Each chain is a contour candidate.

2. **Loop detection** (`core/grouping/loop_detector.py`): verifies which contours form closed
   topological cycles — i.e., the chain returns to its start vertex. Closed loops are candidate
   enclosed regions.

3. **Concentric grouping** (`core/grouping/concentric_grouping.py`): groups CIRCLE and ARC
   entities that share the same center within the 4-decimal-place vertex precision. These
   concentric groups are the primary geometric evidence for bore and shaft features.

4. **Region analysis** (`core/grouping/region_analyzer.py`): identifies disconnected topology
   islands — groups of entities with no topological path between them.

5. **Contour hierarchy** (`core/grouping/contour_hierarchy.py`): for closed loops, determines
   containment by bounding box. Produces `nesting_depth` and `contour_role` (outer / inner).
   Inner loops nested inside the outermost loop are pocket candidates.

---

### Phase 4: Feature Candidate Detection

**Module:** `pipeline/feature_pipeline.py`

Five sub-stages:

1. **Hole candidate detection** (`core/features/hole_candidate_detector.py`): each concentric
   group -> one hole candidate with a radius list, center, and evidence type.

2. **Slot candidate detection** (`core/features/slot_candidate_detector.py`): identifies
   elongated closed contours as slots by bounding-box aspect ratio. Threshold is hardcoded at
   **2.0** — not read from `thresholds.yaml`. Arc-ended contours -> standard slots;
   all-straight-edge contours -> rectangular pockets.

3. **Radial pattern detection** (`core/features/radial_pattern_detector.py`): groups hole
   candidates that are evenly spaced angularly around a common center. Minimum 3 holes required
   to form a pattern.

4. **Symmetry analysis** (`core/features/symmetry_analyzer.py`): detects horizontal and
   vertical mirror symmetry in entity distributions. Produces `mirror_axis` and `axis_position`
   for each symmetry found.

5. **Feature-region grouping** (`core/features/feature_region_grouper.py`): associates feature
   candidates with the topology regions they occupy.

---

### Phase 5: Feature Refinement

**Module:** `pipeline/refinement_pipeline.py`

Five sub-stages:

1. **Confidence analysis** (`core/features/candidate_confidence_analyzer.py`): scores the
   structural reliability of each candidate based on evidence quality.

2. **Hierarchy building** (`core/features/feature_hierarchy_builder.py`): establishes
   parent-child relationships between features (e.g., a hole group inside an outer contour).

3. **Conflict detection** (`core/features/candidate_conflict_resolver.py`): flags candidates
   that claim overlapping geometry. Conflicts are logged but not forcibly resolved — both
   candidates continue downstream.

4. **Repetition consolidation** (`core/features/repeated_pattern_consolidator.py`): groups
   candidates sharing the same geometric signature (type + rounded dimensions) into
   `repetition_groups`.

5. **Ambiguity tracking** (`core/features/structural_ambiguity_tracker.py`): tags candidates
   below the confidence threshold as `ambiguous`. Ambiguity is propagated into training context
   as-is — the pipeline does not resolve or discard ambiguous candidates.

---

### Phase 6: Context Organization

**Module:** `pipeline/context_pipeline.py`

Five sub-stages:

1. **Relationship building** (`core/features/candidate_relationship_builder.py`): produces
   `repetition_sibling`, `concentric`, and `hierarchical_child` relationships from Phase 5
   outputs.

2. **Context cluster analysis** (`core/features/context_cluster_analyzer.py`):
   connected-component analysis on the relationship graph to identify feature clusters.

3. **Relationship confidence** (`core/features/relationship_confidence_manager.py`): scores
   each relationship by the average confidence of its constituent candidates.

4. **Structural dependency mapping** (`core/features/structural_dependency_mapper.py`):
   identifies which features are geometrically dependent on others.

5. **Contextual ambiguity propagation** (`core/features/contextual_ambiguity_propagator.py`):
   candidate-level ambiguity propagates upward to the relationship and cluster levels.

### Phase 7: Semantic Records and Dataset Export

This is the most complex phase. It runs in two parts.

[source: `pipeline/dataset_pipeline.py`, full file — 2196 lines]

#### 7a. Per-Drawing: DatasetPipeline

`DatasetPipeline.run()` runs five stages for each DXF:

[source: `pipeline/dataset_pipeline.py`, `DatasetPipeline.run()`, L45-145]

1. **Supervision mapping** (`core/supervision/supervision_mapper.py`): associates DIMENSION
   and TEXT entities with geometry entities via two strategies:
   - Target-point proximity: DIMENSION `target_points` must be within
     `ASSOCIATION_TOLERANCE = 1.0` unit of a geometry endpoint or circle center.
   - Value comparison: dimension numeric value must be within 1% of a computable geometric
     measurement (line length, diameter, etc.).
   [source: `core/supervision/supervision_mapper.py`, L29]

2. **Context packaging** (`core/supervision/context_packager.py`): for each feature candidate,
   assembles the training context: topology neighbors, neighbor dimensions, contour hierarchy
   position, feature type, repetition groups, concentric group membership.

3. **Target construction** (`core/supervision/target_constructor.py`): selects the primary
   engineering parameter for the feature type as the prediction target. Rejects any target
   whose value is below `min_target_value = 0.1`.
   [source: `dataset_pipeline.py`, L91]

4. **Inference conditioning** (`core/supervision/inference_conditioner.py`): masks the target
   value from the context to prevent direct copying.

5. **Sample assembly** (`core/supervision/sample_assembler.py`): combines the conditioned
   context and target into a training sample tagged `leakage_free: true`.

Output: `data/intermediate/{run_id}/phase7_dataset/{drawing_id}_dataset.json` per drawing.

#### 7b. Batch Export: DatasetExporter + SemanticPipeline

`DatasetExporter.export()` is called once on the full batch of all drawing results.

[source: `pipeline/dataset_pipeline.py`, `DatasetExporter.export()`, L209-403]

**Step 1 — Build semantic records.** For each drawing result, calls
`SemanticPipeline.run()`.

**SemanticPipeline** (`pipeline/semantic_pipeline.py`, 2091 lines):

[source: `pipeline/semantic_pipeline.py`, `SemanticPipeline.run()`, L2000-2074]

`SemanticPipeline.run()` executes in order:
1. `reconstruct_dimensions()` — overall bounding box from entity geometry + dimension text.
2. `map_features()` — maps all structural/feature evidence to `FeatureInstance` objects.
3. `map_relationships()` — maps Phase 6 relationships to `Relationship` objects.
4. `determine_part_type()` — scores the drawing against 8 part classes, picks the highest.
5. Assembles a `SemanticRecord` dataclass.
6. `_validate()` — checks `drawing_id` non-empty, `part_type` non-empty, every feature has
   a non-empty `feature_class` string and a dict `parameters`.

**SemanticRecord schema:**

[source: `pipeline/semantic_pipeline.py`, L84-107]

```python
@dataclass
class SemanticRecord:
    drawing_id: str
    part_type: str
    overall_dimensions: Dict[str, float]   # keys: width, height
    features: List[FeatureInstance]
    relationships: List[Relationship]
    hierarchy: Optional[Dict]
    metadata: Optional[Dict]
```

`FeatureInstance.parameters` is a free-form `Dict[str, Any]`. There is no per-feature-class
typed schema.

**Step 3 — Deduplicate and filter.** Three filters run on the raw task list before balancing:

[source: `dataset_pipeline.py`, L341-366, L832-870]

1. **Deduplication**: exact match on `(drawing_id, json(context), json(target))` triplet.
2. **Empty-context filter**: rejects tasks where all of `feature_parameters_visible`,
   `neighbor_dimensions`, `topology_neighbors`, `relationships`, `symmetries`, and
   `concentric_features` are empty or absent.
3. **Weak-supervision quality check** (above).

**Step 4 — Balance.** `_balance_tasks()`:

[source: `dataset_pipeline.py`, L1690-1737]

```python
counts  = [len(tasks_by_family[f]) for f in tasks_by_family]
median  = float(np.median(counts))
p75     = float(np.percentile(counts, 75))
threshold = int(max(median, p75))
```

For any family exceeding the threshold:
1. **Anchor pass** — include exactly one task per unique `drawing_id` in the family.
2. **Fill pass** — stride over the remaining tasks to fill the quota up to `threshold`.

The threshold is **not fixed**. It is recomputed from the actual family distribution on every
run. The prior documentation's claim of "101" was not a fixed parameter and is not applicable
to the current run.

**Step 5 — Split.** `_split_deterministic()`:

[source: `dataset_pipeline.py`, L1739-1804]

Split ratios: **70% train / 15% validation / 15% test**.

[source: `dataset_pipeline.py`, L29-31]

Algorithm:
1. Strip variant suffixes from drawing IDs: `Corrected_`, `_variant`, `_stress`, `_step`,
   `_v2`, `_200mm`.
2. Group cleaned base IDs by family prefix (leading alphabetic tokens from the filename).
3. For each base ID, compute `MD5("cad_engine_v2_split_salt_" + base_id) % 100`:
   - 0-69 â†’ train
   - 70-84 â†’ validation
   - 85-99 â†’ test
4. Assert that the resulting `train_drawings`, `val_drawings`, `test_drawings` sets have
   zero pairwise intersection. Raises `AssertionError` if any overlap is detected.

All tasks from a given drawing always land in the same split. Split is at drawing level.

**Step 6 — Write JSONL.** `_write_jsonl()`:

[source: `dataset_pipeline.py`, L1806-1954, L1956-1968]

For each task, calls `_build_instruction_prompt()` to render `system`, `user`, `assistant`
strings from the structured context and target. The `task_type` key is then deleted from the
dict, and the remaining fields (including the rendered prompt) are written as one JSON line.

**Exported record structure:**

```json
{
  "drawing_id": "Bearing_Housing_BH08",
  "context": {
    "part_type": "bearing_housing",
    "feature_type": "concentric_bore",
```

## Leakage Prevention

Five independent mechanisms prevent the model from seeing its own answer in context:

**1. InferenceConditioner** (`core/supervision/inference_conditioner.py`):
Masks the target value from the context dict during sample assembly, before Phase 7b sees it.

**2. `_mask_context_leakage()`** in `DatasetExporter`:
Recursively walks the context dict and removes banned properties per task family. Also removes
the numeric value itself (and its half, to prevent radius/diameter leakage). Thread values are
replaced with `[THREAD]` in thread-designation strings.

[source: `dataset_pipeline.py`, L1515-1686]

## Semantic Pipeline: Concept Registry and Text Matching

[source: `pipeline/semantic_pipeline.py`, L114-239]

Three module-level structures govern how annotation text is mapped to engineering facts:

**`DIMENSION_EXCLUDE_WORDS`** — a tuple of text patterns that, when present in a DIMENSION or
TEXT entity, prevent that entity from being classified as a primary engineering dimension.
Examples: `"PCD"`, `"SPACING"`, `"BORE LOC"`, `"TYP"`, `"REF"`.

**`SYNONYMS`** — maps canonical concept names to recognized annotation text aliases:

| Concept | Recognized text patterns |
|---------|--------------------------|
| `THREAD` | THREAD, NPT, BSP, BSPT, UNC, UNF, UNEF, TAPPED, TAP, M-SERIES, THD, THREADED |
| `WEB_THICKNESS` | WEB THK, WEB THICKNESS, WEB THICK, WEAKENED WEB |
| `FLANGE_THICKNESS` | FLANGE THK, FLANGE THICKNESS |
| `ACROSS_FLATS` | AF, A/F, ACROSS FLATS, WIDTH ACROSS FLATS |
| `BEND_RELIEF` | RELIEF, BEND RELIEF, CORNER RELIEF, HINGE RELIEF |

**`CONCEPT_REGISTRY`** — a list of `EngineeringConcept` objects (dataclass: `name`, `synonyms`,
`feature_class`) for: `THREAD`, `BORE`, `HOLE`, `POCKET`, `RIB`, `PORT`, `CHANNEL`,
`SHOULDER`, `COPE`, `CHAMFER`, `RELIEF`, `FIN`, `FLANGE`, `WEB`, `O_RING`.

All text matching uses `_match_keyword()`, which applies the anchored regex
`(?:^|[^A-Z0-9])KEYWORD(?:$|[^A-Z0-9])` to prevent false matches (e.g., the word "PITCH"
would not match a keyword "ITCH").

## Validation Checks at Runtime

Three post-processing checks run inside `DatasetExporter.export()` after semantic records are
built but before tasks are generated:

**1. Outer-dims validation:**
Compares `semantic_record.overall_dimensions` (text-derived) against `reconstruct_dimensions()`
(geometry-derived physical bounding box). Logs a warning if any dimension differs by more than
1 unit AND more than 5% of the physical value. Does not reject the record.

[source: `dataset_pipeline.py`, L263-276]

**2. Concentric bore sanity:**
For every `concentric_bore` feature, asserts `bore_diameter < outer_diameter`. Logs a warning
per violation. Does not reject the record.

[source: `dataset_pipeline.py`, L278-290]

**3. Semantic coverage audit:**
`_generate_semantic_coverage_audit()` iterates every semantic fact (every parameter of every
feature in every semantic record) and maps it to its status in the generated task set:

- `TARGET` — appears as a prediction target in at least one training task
- `CONTEXT` — appears in the context of at least one training task
- `TARGET & CONTEXT` — appears in both roles
- `LOST` — appears in no task in any role

Written to `phase7_export/semantic_coverage_audit.json` after every run.

[source: `dataset_pipeline.py`, L2029-2195]

## How to Add a New Drawing

1. Write a generator script following the existing pattern. Use layers `GEOMETRY`,
   `DIMENSIONS`, `CENTERLINES`. Validate the output with the `CADValidator` class pattern.
2. Run the generator to produce a `.dxf` file.
3. Place the file in `data/raw_dxf/`.
4. Run `python main.py`. The pipeline globs `data/raw_dxf/*.dxf` — no registration,
   no config change, no code edit required to pick up the new file.
