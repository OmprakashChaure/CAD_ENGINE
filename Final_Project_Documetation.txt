## 1. PROJECT OVERVIEW

### Purpose

CAD_ENGINE extracts engineering meaning from DXF (Drawing Exchange Format) technical drawings and produces a structured supervised training dataset for fine-tuning Large Language Models (LLMs) to reason about engineering dimensions.

A DXF file stores raw geometry — lines, circles, arcs, polylines, and text annotations — but contains **no semantic meaning**. A circle in a DXF drawing does not declare whether it is a bore, a clearance hole, a shaft outer diameter, or a thread callout. CAD_ENGINE resolves this by applying a deterministic 7-phase processing pipeline that reconstructs the engineering intent from the geometric evidence.

### Key Objectives

| Objective | Status |
|---|---|
| Parse all 143 DXF engineering drawings across 24 engineering families | **Complete** |
| Extract structured semantic knowledge (features, dimensions, relationships) | **Complete** |
| Generate 732 balanced, leakage-free LLM fine-tuning tasks | **Complete** |
| Achieve 100% drawing coverage with zero cross-split data leakage | **Complete** |
| Fine-tune Qwen2.5-Coder-7B-Instruct on the generated dataset | **Pending** |

### Final Outcomes

- **143 DXF drawings** processed across 24 engineering families (Aerospace, Bearing Housing, Structural Profiles, Kinematic Linkage, Gasket, Gear, Hardware, Thermal, PCB, Fluid Fittings, and more)
- **143 semantic records** generated (`semantic_records.json`, 438 KB)
- **732 training tasks** across 11 task families, split into:
  - `train.jsonl` — 486 tasks
  - `validation.jsonl` — 141 tasks
  - `test.jsonl` — 105 tasks
- **Zero cross-split leakage** — verified by drawing-level isolation
- **100% drawing coverage** — every drawing contributes at least one training task

## 3. DXF EXTRACTION PIPELINE

### Library Used

**`ezdxf`** — Python DXF read/write library (MIT licensed).  
Accessed via: `core/reader/dxf_loader.py` → `ezdxf.readfile(path)`  
Modelspace: `document.modelspace()` in `core/reader/entity_iterator.py`

### Supported Entity Types

```python
# core/reader/entity_iterator.py
SUPPORTED_TYPES = {
    "LINE",        # Straight geometry (structural outlines, webs)
    "CIRCLE",      # Full circles (holes, bores, shafts)
    "ARC",         # Partial arcs (fillets, rounded profile ends)
    "LWPOLYLINE",  # Lightweight polyline (most closed profiles)
    "POLYLINE",    # Legacy polyline format
    "SPLINE",      # Curves — quarantined (no geometry support yet)
    "TEXT",        # Single-line text annotations
    "MTEXT",       # Multi-line text annotations
    "INSERT",      # Block references — quarantined
    "DIMENSION",   # Dimension annotations (PRIMARY supervision source)
    "HATCH",       # Fill patterns — quarantined
}
```

```json
    "numeric_value": 10.0

  "supported": true

```

### Special Dimension Parsing

Dimensions are extracted using three fallback strategies in `GeometryNormalizer._normalize_dimension()`:

1. **Fractional value parser** — handles inch fractions: `"1/2-13 UNC"` → `0.5`
2. **`actual_measurement` attribute** — raw DXF measurement field
3. **Regex text extraction** — strips count prefixes (`"4X "`) and parses first numeric token

Text role classification for TEXT/MTEXT:
- Starts with `M` + digit → `thread_callout`
- Contains `±` → `tolerance`
- Contains `Ø` or `ø` → `diameter_value`
- Starts with `R` + digit → `radius_value`
- Contains `°` → `angle_value`

## 4. PREPROCESSING AND FEATURE ENGINEERING

### 4.1 Filter Chain (Phase 1)

Five sequential filters clean the raw entity list:

| Filter | File | Action |
|---|---|---|
| `TextFilter` | `core/filters/text_filter.py` | Separates text/annotation from geometry |
| `DegenerateFilter` | `core/filters/degenerate_filter.py` | Removes zero-length lines, zero-radius circles, `supported: False` entities |
| `DuplicateFilter` | `core/filters/duplicate_filter.py` | Removes geometrically identical entities (exact coordinate match) |
| `LayerFilter` | `core/filters/layer_filter.py` | Enforces allowed layers: GEOMETRY, DIMENSIONS, CENTERLINES |
| `BorderFilter` | `core/filters/border_filter.py` | Detects and removes title block/drawing border rectangles |

Each filter produces:
- `kept_entities` — entities that pass
- `quarantined_entities` — entities that fail but are preserved
- `removed_entities` — permanently discarded entities
- `statistics` — count report per filter stage

### 4.2 Topology Analysis (Phase 2)

**Purpose:** Determine which geometry entities are physically connected (share endpoints).

**Algorithm (`core/grouping/vertex_indexer.py`):**
- Extract all endpoints from every entity
- Round to 4 decimal places
- Group entities by shared vertex key
- Result: `{vertex_key → [entity_id_1, entity_id_2, ...]}`

**Adjacency construction (`core/grouping/adjacency_builder.py`):**
- Build `{entity_id → [connected_entity_ids]}`
- Hub limit: max 8 connections per vertex (avoids false T-junction associations)

**Engineering use:** Without topology, closed profiles cannot be detected. A bearing housing outer ring is identified as a "closed contour" only because all its LINE and ARC entities form a connected cycle.

**Thresholds (`configs/thresholds.yaml`):**
```yaml
geometry:
  endpoint_snap_tolerance: 0.01
  concentric_tolerance: 0.001
relationships:
  max_hub_connections: 4
```

### 4.3 Structural Recognition (Phase 3)

Five sub-stages produce the structural geometry understanding:

| Stage | Module | Output |
|---|---|---|
| Contour extraction | `contour_extractor.py` | Ordered chains of connected entities |
| Loop detection | `loop_detector.py` | Verified closed topological cycles |
| Concentric grouping | `concentric_grouping.py` | Groups of circles/arcs sharing a center |
| Region analysis | `region_analyzer.py` | Disconnected topology islands |
| Contour hierarchy | `contour_hierarchy.py` | Outer/inner containment with nesting depth |

**Key engineering signal — contour hierarchy:**
- A closed loop *inside* another closed loop → `contour_role: inner`, `nesting_depth: 1`
- The inner loop is a candidate pocket or bore
- An outer loop with no parent → `contour_role: outer`, `nesting_depth: 0`

### 4.4 Feature Candidate Detection (Phase 4)

**Hole candidates (`core/features/hole_candidate_detector.py`):**  
Concentric circle groups → hole candidates. Each candidate carries: `center`, `radius`, `candidate_type` (`single_circle`, `concentric_pair`, `multi_radius`).

**Slot candidates (`core/features/slot_candidate_detector.py`):**  
Closed contours with bounding-box aspect ratio > 2.0 → slot candidates. Contains: `width`, `length`, `area`, `candidate_type` (`slot`, `pocket`, `rectangular_pocket`).

**Radial patterns (`core/features/radial_pattern_detector.py`):**  
Hole candidates spaced at equal angular intervals around a common center → bolt circle / radial pattern. Requires ≥ 3 holes.

**Symmetry analysis (`core/features/symmetry_analyzer.py`):**  
Detects mirror and rotational symmetry. Used as context evidence for repeated features.

### 4.5 Feature Refinement (Phase 5)

| Stage | Module | What It Adds |
|---|---|---|
| Confidence scoring | `candidate_confidence_analyzer.py` | Reliability score per candidate |
| Hierarchy building | `feature_hierarchy_builder.py` | Parent-child relationships between features |
| Conflict detection | `candidate_conflict_resolver.py` | Flags overlapping candidate claims |
| Repetition consolidation | `repeated_pattern_consolidator.py` | Groups identical features into `rep_xxxxx` with `repetition_count` |
| Ambiguity tracking | `structural_ambiguity_tracker.py` | Preserves — never forces — uncertain classifications |

**Key design principle:** Ambiguity is preserved, not resolved. A feature with insufficient structural evidence is tagged as `ambiguous`, and this ambiguity propagates into training context. The model learns that ambiguous geometry requires more careful reasoning.

### 4.6 Context Organization (Phase 6)

Builds the relational structure between features:
- **`repetition_sibling`** — two features have the same geometric signature (e.g., 4 identical slots)
- **`concentric`** — features share a center (bore inside a shaft OD)
- **`hierarchical_child`** — pocket inside main body
- **Context clusters** — connected components of the feature relationship graph

### 4.7 Feature Engineering for Model Input

The final training context for each task contains:

```json
"context": {
  "feature_type": "concentric_bore",
  "topology_neighbors": ["ent_00003", "ent_00015"],
  "neighbor_dimensions": [
    {"neighbor_id": "ent_00003", "dimension_type": "radius", "value": 35.0}
  ],
  "hierarchy": {
    "contour_role": "inner",
    "nesting_depth": 1,
    "child_count": 0,
    "has_parent": true
  },
  "feature_context": {"candidate_id": "hc_00001", "candidate_type": "concentric"},
  "repetition_context": {"group_id": "rep_00001", "repetition_count": 4, "signature": "slot:(4.0, 17.0)"},
  "concentric_context": {"group_id": "conc_00001", "radii": [25.0, 35.0], "count": 2},
  "relationships": [
    {"source_candidate_id": "hc_00001", "target_candidate_id": "hc_00002",
     "relationship_type": "repetition_sibling", "context": "rep_00001"}
  ],
  "feature_id": "concentric_bore_1",
  "feature_parameters_visible": {"outer_diameter": 70.0}
}
```

**Leakage prevention:** `InferenceConditioner` (`core/supervision/inference_conditioner.py`) strips the target property from `feature_parameters_visible` before the task is written. `_mask_context_leakage()` in `dataset_pipeline.py` verifies this. Every sample carries a `leakage_free: true` marker.

## 8. CHALLENGES AND SOLUTIONS

### Challenge 1: Dimension-to-Geometry Association

**Problem:** DXF DIMENSION entities record where dimension lines touch the drawing (via `defpoint2`–`defpoint5`), but there is no explicit link to the geometry being dimensioned. A "50mm" dimension could belong to any 50mm-long line in the drawing.

**Solution (`core/supervision/supervision_mapper.py`):**  
Two-strategy matching:
1. **Target-point proximity** — match dimension defpoints to entity endpoints/centers within 1.0 unit tolerance
2. **Value comparison** — compare dimension value to all computable geometry measurements within 1% tolerance

```python
ASSOCIATION_TOLERANCE = 1.0  # units

def _match_by_target_points(target_points, geometry_ents):
    # Find geometry entity closest to dimension's target points
    ...

def _match_by_value(sup_value, computable, geometry_ents):
    # Match within 1% of computable geometry measurement
    ratio = abs(sup_value - comp_value) / comp_value
    if ratio < 0.01:
        return comp["entity_id"]
```

### Challenge 2: Fractional and Non-Standard Thread Dimensions

**Problem:** British pipe threads (BSP) and inch-fraction dimensions appear as text strings: `"G 1/4"`, `"1/2-13 UNC"`, `"M10×1.5"`. These cannot be parsed as plain floats.

**Solution (`core/classifiers/geometry_normalizer.py`):**

```python
def parse_fractional_value(text: str):
    match = re.search(r"(?:(\d+)[- ])?(\d+)/(\d+)", text)
    if match:
        whole = float(whole_str) if whole_str else 0.0
        return whole + float(num) / float(den)
```

Thread callouts are preserved as **string targets** (not floats) in the dataset. The model learns `"M10×1.5 TAPPED"` is the answer to `infer_thread_size` — exactly as an engineer would write it.

### Challenge 3: ST14 Coverage Loss (Balancing Bug)

**Problem:** `Structural_ST14_BulbFlat` had 2 valid tasks (both in the `infer_feature_span` family, which had 113 tasks total). The stride-based downsampling of this family to 101 tasks happened to skip both ST14 tasks.

**Root cause (`pipeline/dataset_pipeline.py`, old code):**
```python
# Old: pure stride — no drawing protection
stride = len(family_tasks) / threshold
for i in range(threshold):
    selected.append(family_tasks[int(i * stride)])  # ST14 tasks fell between strides
```

**Solution (implemented in `_balance_tasks()`):**
```python
# New: anchor pass — guarantee ≥1 task per drawing
for d in unique_drawings:
    selected.append(drawings_map[d][0])   # ← ST14 protected here

# Fill pass — stride on remaining slots
remaining = [t for d in unique_drawings for t in drawings_map[d][1:]]
needed = threshold - len(selected)
stride = len(remaining) / needed
for i in range(needed):
    idx = int(round(i * stride))
    selected.append(remaining[idx])
```

**Result:** ST14 now contributes 2 tasks to train. 143/143 drawing coverage achieved.

### Challenge 4: Task Family Imbalance

**Problem:** `infer_profile_dimension` produced 218 tasks vs `infer_thread_size` with 28. A naïve fixed cap (e.g., 100) would either waste data or make the cap too aggressive.

**Solution (`_balance_tasks()` adaptive threshold):**
```python
import numpy as np
median = float(np.median(counts))   # ≈ 77 in final run
p75    = float(np.percentile(counts, 75))  # ≈ 101
threshold = int(max(median, p75))          # → 101
```

Families > 101 are downsampled; families ≤ 101 are left intact. Result: balanced representation without artificially limiting smaller families.

### Challenge 5: Cross-Split Data Leakage

**Problem:** If the same drawing contributes tasks to both train and test, the model can memorize drawing-specific geometry rather than learning general engineering reasoning.

**Solution (`_split_deterministic()`):**
- Assignments are made at **drawing level**, not task level
- MD5 hash of `base_drawing_id` → deterministic 0–99 bucket
- Verified with explicit assertion:
  ```python
  assert len(train_drawings & val_drawings) == 0
  assert len(train_drawings & test_drawings) == 0
  assert len(val_drawings & test_drawings) == 0
  ```

### Challenge 6: Pocket vs. Bore Ambiguity

**Problem:** An inner closed loop could be a bore (circular feature), a pocket (milled region), or a slot (elongated cut). All three are valid inner contours. Forcing a classification risks mislabeling.

**Solution:** Preserve structural evidence and let the semantic pipeline decide:
1. `hole_candidate_detector.py` catches circles / concentric pairs → bore evidence
2. `slot_candidate_detector.py` catches aspect ratio > 2.0 → slot evidence
3. `contour_hierarchy.py` catches nesting → pocket evidence
4. `structural_ambiguity_tracker.py` flags cases where multiple evidence types exist
5. Context packager packages ALL available evidence into the training context
6. The model learns from this rich evidence instead of a premature label
